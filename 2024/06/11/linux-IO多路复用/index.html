IO多路复用

IO多路复用（Input/Output Multiplexing）是一种在单个线程中管理多个输入/输出通道的技术2。它允许一个线程同时监听多个输入流（例如网络套接字、文件描述符等），并在有数据可读或可写时进行相应的处理，而不需要为每个通道创建一个独立的线程。常见的IO多路复用机制包括select、poll和epoll，这些机制通过将多个IO通道注册到一个事件管理器中，然后通过阻塞方式等待事件的发生

IO多路复用的工作原理是通过一种机制，允许单个进程监视多个文件描述符，一旦某个文件描述符就绪（例如，一个网络连接上有数据可读），就能够通知程序进行相应的读写操作。

这里的“多路”指的是多个网络连接或文件流，“复用”指的是使用同一个进程或线程来处理多个流。

在传统的阻塞IO模型中，每个IO操作通常需要一个独立的线程来处理，这在处理大量并发连接时会导致线程数量的增加，从而带来线程切换和上下文切换的开销。而IO多路复用通过使用一个线程来监听多个IO事件，避免了线程数量的增加，减少了这些开销

常见的IO多路复用技术包括select、poll和epoll

  • select和poll：这两种机制都是通过告诉内核监视一系列的文件描述符，然后进程阻塞并等待内核通知它哪些文件描述符就绪了。select有最大文件描述符数量的限制，而poll没有这个限制。但它们都有一个共同的缺点，那就是当监视的文件描述符数量很多时，每次调用都需要遍历整个集合来检查状态，效率较低1。

  • epoll：与select和poll不同，epoll是在Linux 2.6及以上版本中提供的一种高效的IO事件通知机制。它不需要遍历整个文件描述符集合，而是当文件描述符状态发生变化时，就会将这个事件通知给应用程序。epoll可以处理数以万计的并发连接，而且效率不会随着文件描述符数量的增加而降低。epoll的另一个特点是它支持“边缘触发”和“水平触发”，前者只在状态变化时通知一次，而后者只要条件满足就会不断通知1

虽然epoll在处理大量并发连接时比select和poll更高效，但这并不意味着select和poll就完全没有用处了。select和poll在某些情况下仍然有其适用性和优势。

select&poll

select与poll的逻辑相似，都是把要监听的socket存入一个数据结构中（select-bitmap）然后每次发生事件后循环遍历这个数据结构返回发生事件的socket，这里的事件可能是读事件/写事件。都是水平触发( level-trigger)

读事件：1）已连接队列中有已经准备好的socket（有新的客户端连上来了）；
2）接收缓存中有数据可以读（对端发送的报文已到达）；
3）tcp连接已断开（对端调用close()函数关闭了连接）。
写事件：发送缓冲区没有满，可以写入数据（可以向对端发送报文）。

select

select 的实现思路很直接，假如程序同时监视sock1、sock2 和 sock3 三个socket，那么在调用select 之后，操作系统把进程分别加入这三个socket的等待队列中. 每次调用select都需要将进程加入到所有监视socket的等待队列，每次唤醒都需要从每个队列中移除。这里涉及了两次遍历，而且每次都要将整个 FDS 列表传递给内核，有一定的开销.正是因为遍历操作开销大，出于效率的考量，才会规定select 的最大监视数量，默认只能监视1024个socket。

epoll

epoll是在select 出现 N 多年后才被发明的，是select 和 poll的增强版本。epoll通过以下一些措施来改进效率：

1.功能分离

​ select 低效的原因之一是将“维护等待队列”和“阻塞进程”两个步骤合二为一。

epoll将这两个操作分开，先用epoll_ctl 维护等待队列，再调用epoll_wait 阻塞进程。显而易见，效率就能得到提升。

2.就绪列表

select低效的另一个原因在于程序不知道哪些socket收到数据，只能一个个遍历。如果内核维护一个“就绪列表”，引用收到数据的socket，就能避免遍历。

实现

当某个进程调用epoll_create方法时，内核会创建一个 eventpoll 对象（也就是程序中 epfd 所代表的对象）。eventpoll 对象也是文件系统中的一员，和socket一样，它也会有等待队列。

创建一个代表该epoll的 eventpoll 对象是必须的，因为内核要维护“就绪列表”等数据，“就绪列表”可以作为 eventpoll 的成员。epoll使用双向链表来实现就绪队列，使用红黑树作为索引结构

创建epoll对象后，可以用epoll_ctl 添加或删除所要监听的socket。以添加socket为例。

当socket收到数据后，中断程序会操作 eventpoll 对象，而不是直接操作进程。

当socket收到数据后，中断程序会给 eventpoll 的“就绪列表”添加socket引用。

eventpoll 对象相当于socket和进程之间的中介，socket的数据接收并不直接影响进程，而是通过改变 eventpoll 的就绪列表来改变进程状态。

当程序执行到epoll_wait 时，如果rdlist 已经引用了socket，那么epoll_wait 直接返回，如果 rdlist 为空，阻塞进程。

在实际的使用中，IO多路复用要求不能阻塞在任意环节，所以应该设置非阻塞

整体逻辑：将所有需要epoll监控的socket通过epoll_event（这个结构体更是为了设置socket的一个临时方法，并不存储实际事件，只用来保存当前要进行设置的fd）来设置其需要监视的事件类型（读/写）以及触发方式（边缘触发/水平触发），然后通过epoll_ctl指定EPOLL_CTL_ADD来加入到epoll的fd中，然后就可以用过epoll_wait开始监视事件，并返回发生事件的socket到第epoll_wait的二个参数中，处理事件后使用epoll_ctl指定EPOLL_CTL_DEL删除对应的socket。
