互斥锁

发表于 2024-06-22 更新于 2024-06-27 分类于 C++ ， Multi-threaded 阅读次数：
本文字数： 4.6k 阅读时长 ≈ 4 分钟

mutex

一个同步原语，用于保护共享数据不被多个线程同时访问。它提供了独占的、非递归的所有权语义，确保同一时刻只有一个线程可以访问被保护的数据。

是用于保护共享数据免受多个线程同时访问的同步原语。互斥量提供独占的、非递归的所有权语义：调用线程从成功调用lock或try_lock开始拥有互斥量，直到调用unlock为止

基本使用

包含头文件：#include <mutex>

  1. 创建互斥量：

    std::mutex mtx;

  2. 加锁和解锁：

    mtx.lock(); // 加锁

    // 访问共享资源

    mtx.unlock(); // 解锁

C++11还提供了三种

  • l timed_mutex：带超时机制的互斥锁。

  • l recursive_mutex：递归互斥锁。

  • l recursive_timed_mutex：带超时机制的递归互斥锁。

锁

std::mutex

  1. 加锁 lock()

    互斥锁有锁定和未锁定两种状态。

    如果互斥锁是未锁定状态，调用lock()成员函数的线程会得到互斥锁的所有权，并将其上锁。

    如果互斥锁是锁定状态，调用lock()成员函数的线程就会阻塞等待，直到互斥锁变成未锁定状态。

  2. 解锁unlock()

    只有持有锁的线程才能解锁。

  3. 尝试加锁try_lock()

    如果互斥锁是未锁定状态，则加锁成功，函数返回true。

    如果互斥锁是锁定状态，则加锁失败，函数立即返回false。（线程不会阻塞等待）

代码中我们有一个全局变量g_value，在task的循环中进行+1-1，如果我们让其多线程运行

正常情况下的期望结果是0,但是由于两个线程同时访问一个变量导致竞态条件发生，所以结果并不是0，这时候我们的解决办法之一就是mutex对加减操作进行加锁。

lock和unlock中的部分是临界区资源，互斥锁就是对临界区资源的保护，互斥锁（mutex）的本质是通过PV操作（即信号量的加减）来实现对临界区资源的保护。

关键点总结

  1. 互斥锁的操作对象：lock和unlock操作的是互斥锁本身（即mtx），这些操作改变了互斥锁的状态（锁定或解锁）。
  2. 保护临界区资源：通过对互斥锁的操作，间接地控制了对临界区资源的访问。也就是说，互斥锁的状态决定了是否允许线程进入临界区。

互斥锁的工作原理

  • 加锁（lock）：当一个线程调用lock()时，如果互斥锁是未锁定状态，线程将成功获取锁并进入临界区；如果互斥锁是锁定状态，线程将被阻塞，直到锁被释放。
  • 解锁（unlock）：当一个线程调用unlock()时，它会释放互斥锁，使其他被阻塞的线程有机会获取锁并进入临界区。

也就是当前线程拥有这个锁后，就可以使用这个锁对选定的部分操作进行上锁操作。

锁管理

  • std::lock_guard：简单的RAII锁管理器，适用于需要在整个作用域内锁定一个互斥锁的情况。
  • std::scoped_lock：可以同时锁定多个互斥锁，避免死锁，适用于需要同时锁定多个互斥锁的情况。
  • std::unique_lock：提供更灵活的锁管理功能，如延迟加锁、提前解锁等，适用于需要在作用域内多次加锁和解锁的情况。

之前mutex部分的代码中，如果在lock之后，程序由于异常/满足某一条件退出程序，就会导致这个锁一直处于上锁的状态，这就可能导致其他尝试获取该锁的线程将被永久阻塞，导致程序无法继续执行（死锁），可能会导致资源泄漏，特别是在长时间运行的程序中，这会影响系统性能。

解决：

  1. 使用std::lock_guard

  2. 使用std::unique_lock

再一种情况：

如果有两个线程分别运行task1和task2，由于上锁顺序，两个线程将互相等待对方所持有的资源导致死锁，要解决这个问题，

  1. 确保所有线程以相同的顺序获取锁。这是最简单和最常见的解决方法。

  2. 使用std::scoped_lock

    C++17引入的std::scoped_lock可以同时锁定多个互斥锁，并且使用了死锁避免算法。

  1. 尝试锁定

    使用try_lock方法尝试获取锁，如果获取失败，可以避免死锁，但需要处理锁定失败的情况。
