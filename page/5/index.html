在C和C++中，main 函数可以有不同的形式，这取决于程序是否需要从命令行接收参数或环境变量。通常，如果程序不需要命令行参数或环境变量，可以使用 int main() 或 int main(void)。如果程序需要命令行参数，可以使用 int main(int argc, char* argv[])，其中 argc 表示参数数量，argv 是参数的数组。

而在vs中不再使用putenv 和 getenv ，理由之后再说/

  1. putenv vs _putenv:
    • putenv 是 POSIX 标准的一部分，用于设置或更改环境变量。在 Unix 类系统（如 Linux）中，putenv 是常用的函数。
    • _putenv 是 Microsoft Visual C++ 提供的函数，它是 putenv 的替代品，用于在 Windows 系统中设置环境变量。它与 ISO C++ 标准兼容，因此在 Windows 平台上编程时推荐使用 _putenv。
  2. getenv vs _dupenv_s:
    • getenv 是 C 和 POSIX 标准库中的一个函数，用于获取环境变量的值。但是，getenv 返回的是指向内部存储的指针，这可能会在多线程环境中引起问题，因为环境变量可能被其他线程更改。
    • _dupenv_s 是一个更安全的函数，它在 Microsoft Visual C++ 中提供。它不仅返回环境变量的值，还会为这个值分配新的内存，避免了 getenv 可能带来的线程安全问题。因此，在需要考虑线程安全的 Windows 程序中，推荐使用 _dupenv_s。

总的来说，_putenv 和 _dupenv_s 是在 Windows 平台上，为了提高安全性和符合标准而推荐使用的函数。它们提供了与 putenv 和 getenv 相同的功能，但是在处理环境变量时更安全，更适合多线程环境

makefile

Makefile 是一个文本文件，用于定义如何编译和链接源代码以生成可执行文件、库文件或其他构建目标。它是 make 工具的输入文件，make 工具根据 Makefile 中定义的规则自动执行编译和链接操作。在 Makefile 中，你可以定义变量、编译规则、文件依赖关系等，以简化和自动化编译过程。例如，你可以指定哪些文件需要编译，哪些文件需要先编译，哪些文件需要后编译，以及哪些文件在源文件更改后需要重新编译123。

Makefile 的语法主要包括以下几个部分：

  1. 目标（Targets）：指明要生成的文件或执行的操作。
  2. 依赖（Dependencies）：目标文件依赖的文件，即在生成目标之前需要先存在或最新的文件。
  3. 命令（Commands）：一系列执行的shell命令，用于生成目标文件。
  4. 变量（Variables）：可以在 Makefile 中定义变量，以便重复使用。
  5. 规则（Rules）：由目标、依赖和命令组成的指令，告诉 make 如何生成一个目标。
  6. 伪目标（Phony Targets）：不是文件的目标，通常用于执行特定的命令序列。

以下是一些具体的语法规则：

  • 规则的基本语法：

    目标...: 依赖... 命令

    每条命令前必须有一个Tab键作为缩进。

  • 变量的使用： 使用 $ 符号表示取变量的值，例如：

    CC = gcc CFLAGS = -g -Wall $(CC) $(CFLAGS) -c main.c

  • 通配符和模式匹配： 使用 % 作为模式匹配，例如：

    %.o: %.c $(CC) -c $< -o $@

  • 函数的使用： Makefile 提供了许多内置函数，例如 wildcard、patsubst 等，用于文件名和字符串的操作。

  • 条件判断： Makefile 支持条件判断语句，如 ifeq、ifneq、ifdef 和 ifndef。

  • 包含其他 Makefile： 使用 include 关键字可以包含其他 Makefile 文件。

  • 注释： 使用 # 开头的行表示注释。

2.条件判断

常见的条件判断语句：

  1. ifeq 和 ifneq：用于比较两个值是否相等或不相等。
  1. ifdef 和 ifndef：用于检查一个变量是否被定义。

在使用条件判断时，需要注意以下几点：

  • ifeq 和 ifneq 后面的参数可以用小括号 () 或者是单引号 '' 包围。

处理子目录之间的依赖关系是 Makefile 中一个重要的高级用法，尤其是在大型项目中。以下是一些处理子目录依赖关系的方法：

  1. 明确声明依赖关系：在主 Makefile 中，您可以明确指出子目录之间的依赖关系，并按照依赖顺序调用子 Makefile。
  2. 使用递归 Make：通过在主 Makefile 中递归地调用 $(MAKE) 命令，可以构建子目录。如果子目录之间有依赖关系，可以在主 Makefile 中按照依赖顺序排列递归调用。
  3. 自动生成依赖关系：对于 C/C++ 项目，您可以使用编译器提供的 -M 选项来自动生成依赖关系文件（如 .d 文件），然后在 Makefile 中包含这些文件来自动处理依赖关系。
  4. 使用非递归 Make：通过在主 Makefile 中包含子目录的 Makefile 片段（如 subdir.mk），可以将整个项目的构建规则集中管理，同时处理依赖关系。
  5. 使用高级构建系统：例如 CMake，它可以帮助管理复杂的依赖关系，并生成适用于不同平台的 Makefile 或其他构建脚本。
  6. VPATH 和 vpath：使用这些变量可以指定额外的搜索路径，这样 make 就可以在这些路径中查找依赖文件。
  7. 条件判断：在子 Makefile 中使用条件判断来确定是否需要重新构建某些目标，这可以基于文件是否存在或某些变量的值

OpenGL

opengl就是一个状态机。

C++和OpenGL之间的关系可以这样理解：

  • C++程序：这是运行在CPU上的程序，它负责执行应用程序逻辑，处理输入输出，以及管理资源等。在OpenGL应用程序中，C++代码用于设置和控制渲染流程，包括调用OpenGL的API来执行具体的图形操作。
  • OpenGL程序：这是运行在GPU上的程序，它由一系列着色器组成，这些着色器是用GLSL（OpenGL Shading Language）编写的。这些着色器程序负责处理所有的图形渲染任务，如顶点变换、光照计算、像素着色等。

当你在C++的main函数中编写OpenGL相关的代码时，你实际上是在使用C++来控制GPU的行为。以下是整个过程的简化版：

  1. 编写着色器代码：在C++程序中，你会编写顶点着色器和片段着色器的源代码。
  2. 创建和编译着色器：通过CreateShader和CompileShader函数，你会创建OpenGL着色器对象，并将你的GLSL代码编译成GPU可以理解的形式。
  3. 链接着色器到程序：编译好的着色器会被附加到一个OpenGL程序对象上，然后这个程序对象会被链接。链接是将不同的着色器合并成一个完整的可执行程序的过程。
  4. 验证程序：链接完成后，程序会被验证，以确保它可以在当前的OpenGL环境中运行。
  5. 使用程序：在C++的main函数中，通过调用glUseProgram，你会告诉GPU使用之前链接好的程序来渲染图形。

所以，你可以将C++程序看作是“指挥官”，它指挥GPU做什么；而OpenGL程序就像是在GPU上执行具体任务的“工人”。在C++程序中准备好一切后，通过glUseProgram实际上启动GPU上的OpenGL程序来进行渲染工作

OpenGL编程的大体流程涉及多个阶段，每个阶段都有其特定的功能和作用。下面是一个简化的流程概述

  1. 创建OpenGL上下文（Context）：这是开始任何OpenGL编程之前的必要步骤。上下文是一个巨大的状态机，它保存了OpenGL中的各种状态。

  2. 设置视口（Viewport）：这一步涉及到定义一个屏幕上的矩形区域，OpenGL会将最终的图像渲染到这个区域内。

  3. 准备着色器和程序（Shaders and Program）：着色器是运行在GPU上的小程序，负责处理图形和像素的渲染。在OpenGL中，至少需要编写并编译顶点和片段着色器，然后将它们链接到一个程序中，这个程序会在渲染过程中使用。

  4. 创建缓冲区（Buffers）：在GPU内存中创建缓冲区来存储顶点数据、颜色、纹理坐标等信息。这些数据会在渲染时被着色器程序使用。

  5. 加载纹理（Textures）：如果需要的话，加载纹理数据到GPU，以便在渲染中应用到3D对象上。

  6. 设置渲染循环（Render Loop）：这是一个持续运行的循环，不断地绘制图像并刷新屏幕。在每次迭代中，会执行清除屏幕、设置摄像机和视图变换、绘制对象等操作。

  7. 输入处理（Input Handling）：响应用户输入，如键盘和鼠标事件，以实现交互式图形应用。

  8. 清除和绘制（Clear and Draw）：在每次迭代中，清除颜色缓冲区和深度缓冲区，然后根据缓冲区中的数据绘制3D对象。

  9. 交换缓冲区（Buffer Swap）：在双缓冲模式下，绘制完成后，将后台缓冲区的内容交换到前台，显示到屏幕上。

关于 准备着色器和程序（Shaders and Program）部分。这段代码主要负责以下几个任务：

  1. 编译着色器（Compile Shaders）：CompileShader函数接收着色器类型和源代码字符串，创建着色器对象，将源代码与着色器关联，然后编译着色器。如果编译失败，它会打印错误信息并返回0。
  2. 链接着色器到程序（Link Shaders to Program）：CreateShader函数创建一个OpenGL程序对象，调用CompileShader函数分别编译顶点和片段着色器，然后将这两个着色器附加到程序对象上。接着，它链接这个程序对象，使其成为一个完整的可执行程序，最后返回程序对象的ID。

这些步骤是设置OpenGL渲染管线的关键部分，确保了GPU能够正确地执行着色器代码来渲染图形。

这里提到的程序是实际上的GPU上运行的程序，当main中写好的着色器代码传入CreateShader后编译这些代码，编译完成后将着色器附加到程序相当于是把编译好的代码添加到程序中，然后链接验证程序，最后将这个程序的id作为返回值返回，也就是相当于在CreateShader 和CompileShader 作准备工作，也就是编译着色器代码和创建程序以及两者的链接，然后在main中调用glUseProgram才会实际上执行这个程序

顶点着色器（可编程）

当今大多数显卡都有成千上万的小处理核心，它们在GPU上为每一个（渲染管线）阶段运行各自的小程序，从而在图形渲染管线中快速处理你的数据。这些小程序叫做着色器(Shader)。

图形渲染管线的第一个部分是**顶点着色器(Vertex Shader)**，它把一个单独的顶点作为输入。顶点着色器主要的目的是把3D坐标转为另一种3D坐标（后面会解释），同时顶点着色器允许我们对顶点属性进行一些基本处理。

它会在GPU上

  1. 创建内存用于储存我们的顶点数据，
  2. 配置OpenGL如何解释这些内存，并且指定其如何发送给显卡。
  3. 顶点着色器接着会处理我们在内存中指定数量的顶点

通过顶点缓冲对象(Vertex Buffer Objects, VBO)管理2中的这个内存，它会在GPU内存（通常被称为显存）中储存大量顶点。使用这些缓冲对象的好处是我们可以一次性的发送一大批数据到显卡上，而不是每个顶点发送一次。从CPU把数据发送到显卡相对较慢，所以只要可能我们都要尝试尽量一次性发送尽可能多的数据。当数据发送至显卡的内存中后，顶点着色器几乎能立即访问顶点，这是个非常快的过程。

顶点数据储存在显卡的内存中，用VBO这个顶点缓冲对象管理。

片段着色器

计算像素最后的颜色输出

在计算机图形中颜色被表示为有4个元素的数组：红色、绿色、蓝色和alpha(透明度)分量，通常缩写为RGBA。当在OpenGL或GLSL中定义一个颜色的时候，我们把颜色每个分量的强度设置在0.0到1.0之间。比如说我们设置红为1.0f，绿为1.0f，我们会得到两个颜色的混合色，即黄色。这三种颜色分量的不同调配可以生成超过1600万种不同的颜色！

在OpenGL中，当你绑定一个IBO（索引缓冲对象）到GL_ELEMENT_ARRAY_BUFFER目标，并且此时有一个VAO（顶点数组对象）处于绑定状态，IBO的绑定就会被VAO记录下来。这意味着，VAO不仅保存了顶点属性指针的状态，也保存了索引缓冲的绑定状态。

这里是发生这一过程的关键步骤：

  1. 创建并绑定VAO。
  2. 绑定VBO并设置顶点属性指针（这些状态被保存在VAO中）。
  3. 绑定IBO到GL_ELEMENT_ARRAY_BUFFER（这个绑定也被保存在VAO中）。

顶点属性和内存布局

在OpenGL中，理解属性（Attributes）如位置（Position）、纹理（Texture）、法线（Normal）是3D图形编程的基础。这些属性通常用于定义一个物体的几何形状和外观，它们在顶点着色器（Vertex Shader）中被处理，并用于渲染管线中的各种计算，**==顶点一词不只是表示一个点，而是顶点相关的所有属性的集合，也就是说一个顶点有多个属性数据==**。

  • 位置（Position）: 通常是一个三维向量，表示顶点在空间中的坐标。在顶点着色器中，位置会被变换到裁剪空间（Clip Space），以便进行视图和投影变换。
  • 纹理（Texture）: 纹理坐标用于将纹理映射到3D对象上。纹理坐标通常是二维的，与纹理图像的坐标系统对应，用于在片段着色器（Fragment Shader）中采样纹理图像。
  • 法线（Normal）: 法线是垂直于顶点表面的单位向量，用于光照计算。在光照模型中，法线用于确定表面与光源之间的角度，从而计算出正确的光照效果。

在实际应用中，法线贴图（Normal Mapping）技术可以用来增强视觉效果，使得平面表面看起来具有凹凸感。法线贴图通过在每个像素处修改法线向量，模拟出更加复杂的表面细节。这需要在切线空间（Tangent Space）中进行计算，其中切线（Tangent）和副切线（Bitangent）向量与法线向量一起定义了一个局部的坐标系。

在使用法线贴图时，需要计算出切线和副切线，然后构建一个TBN矩阵（Tangent-Bitangent-Normal Matrix），这个矩阵用于将法线从切线空间变换到世界空间或视图空间，以便进行光照计算。这个过程涉及到顶点着色器和片段着色器中的一系列向量和矩阵变换。

总的来说，属性、位置、纹理和法线是3D图形中用于定义物体外观和行为的基本元素，它们在OpenGL渲染管线中起着至关重要的作用。通过理解和正确使用这些概念，可以创建出丰富多彩的3D场景和效果。如果你有更具体的问题或需要示例代码，随时可以问我！

std::unordered_map

std::map和std::unordered_map是C++标准库中的两种非常有用的容器，它们都可以存储键值对，但在内部实现和性能特性上有所不同。

std::map:

  • std::map是一个基于红黑树的平衡二叉搜索树实现的关联容器。
  • 它存储的元素是按照键来排序的，因此可以非常快速地进行查找、插入和删除操作。
  • std::map中的键是唯一的，不能有重复的键。
  • 它的主要优点是，可以保证元素按照特定顺序存储，通常是按照键的升序。
  • 查找操作的时间复杂度为O(log n)，其中n是容器中元素的数量。

std::unordered_map:

  • std::unordered_map是基于哈希表实现的关联容器。
  • 它不保证元素的顺序，因此不能依赖于元素的任何排序。
  • std::unordered_map中的键也是唯一的，不能有重复的键。
  • 它的主要优点是，在理想情况下，查找、插入和删除操作的时间复杂度可以接近O(1)。
  • 但是，由于哈希冲突的存在，这些操作的时间复杂度可能会变差，尤其是当哈希函数分布不均匀时。

在选择使用std::map还是std::unordered_map时，需要考虑你的应用场景。如果你需要保持元素的顺序，则应该使用std::map。如果你更关心查找效率，并且不在乎元素的顺序，那么std::unordered_map可能是更好的选择。

std::unordered_map

前面提到了，std::unordered_map就是哈希表，什么是哈希表，看看newbing的解释：

想象一下，你有一个巨大的图书馆，你需要一种方法快速找到任何一本书。如果你把所有的书按照书名的字母顺序排列，那么即使图书馆很大，你也可以通过书名快速找到书。但是，如果每次你想找一本书都要从头到尾查找，那就太慢了。这就是哈希表发挥作用的地方。

哈希表就像是一个有很多小抽屉的巨大书架。每个抽屉都有一个特定的编号，而这个编号是由哈希函数决定的。哈希函数就像是一个魔法转换器，你告诉它你想找的书名，它就给你一个抽屉的编号。这样，你就可以直接去那个编号的抽屉找书，而不用查看每一本书。

你只需要记住这个编号，就可以直接找到书，这样查找速度就非常快了。

但是，有时候会出现这样的情况：两本不同的书名被哈希函数转换成了同一个抽屉编号。这就是所谓的哈希冲突。比如说，”哈利波特”和”指环王”都被转换成了编号5。这时候，我们需要一种方法来解决这个问题，通常的做法是在同一个抽屉里再加一个小盒子，这样”哈利波特”可以放在一个盒子里，”指环王”放在另一个盒子里。

这样，即使两本书的编号相同，你也可以通过小盒子的标记来区分它们。

实际上就是哈希表中的每一个节点是一个键值对，一个键和一个值（key-value pair）通过键映射到值

而中间这个映射过程是哈希函数的工作,哈希函数是一种将输入数据（无论大小）转换成固定大小输出的函数，通常输出是一个整数，这个输出称为哈希值或哈希码。哈希函数的设计目标是尽可能快地计算出哈希值，并且尽量减少不同输入产生相同输出值的情况, 字太多了不想看，以后回来更不想看

这么理解：==哈希函数可以看作是一种自定义的查找规则，它将输入转换成一个哈希值，这个哈希值通常用作在哈希表中查找对应内容的索引。==

  • 对于基本数据类型和标准库中的类型，C++标准库提供了默认的哈希函数。这些默认的哈希函数通常是为了提供一个简单且高效的方式来将数据转换成一个整数值，这个整数值就是哈希值。例如，对于整数类型，哈希函数可能就是简单地返回其本身的值；对于字符串类型，哈希函数会计算字符串中字符的某种组合，以生成一个唯一的整数值。
  • 但是，如果你想在std::unordered_map中使用自定义类型作为键，你需要提供一个合适的哈希函数和相等性判断函数。这通常是通过特化std::hash模板和提供operator==来完成的。

刚开始看到这里已经分不清哈希值和键是什么关系了，在开头的例子中我们已经学会了unordered_map的基本使用，查找或者插入操作在现在这个例子中是这样的：

操作符”[ ]”

写过数据结构的都知道，常常要为一个操作符或函数提供不同的版本来适配const和非const，但是这个方括号操作符并没有const版本，为什么呢？

operator[]：如果键不存在，它会创建一个新的键值对，并将值初始化为该类型的默认值。例如，如果值类型是int，新创建的键值对的值将被初始化为0。const表示常量不可修改，而”[ ]”本身就是修改的行为，所以没有const版本，对应的容器内有个at方法

.at()：这个方法只用来访问已存在的键值对。如果你尝试使用.at()访问一个不存在的键，它会抛出一个std::out_of_range异常。.at()方法不会创建新的键值对因此，你可以认为.at()是一个更安全的方法，因为它不会意外地插入新的键值对。如果你需要确保不修改unordered_map，应该使用.at()方法。

回到之前的内容

当你向std::unordered_map插入一个键值对或者查找一个键时，发生的步骤大致如下：

  1. 哈希函数接收键作为输入。
  2. 哈希函数计算并返回一个哈希值。
  3. 哈希表根据这个哈希值决定元素应该存储或查找的桶的位置。

这个过程使得查找和插入操作非常快速，因为它避免了遍历整个容器。哈希值是内部使用的，而作为用户，你通常不需要直接处理哈希值，你只需要关心键和值。

方才那个hash模板特化的代码属是抽象， hash<>()() 这这这，大不合适，但是也不理解了，会用即可，

  1. hash<int>()和hash<std::string>()是创建一个std::hash类型的临时对象，这个对象专门用来计算int类型和std::string类型的哈希值。
  2. 紧跟在后面的()是对这个临时对象的函数调用操作符，它将 key.CityName 作为参数传入，计算它们的哈希值。

让我们分解一下这个表达式：

  • std::hash<int>()创建了一个用于计算整数哈希值的std::hash对象。
  • std::hash<int>()(key.CityName)调用这个对象的operator()，传入key.CityName作为参数，返回key.CityName的哈希值。

总之：无论是普通类型还是自定义类型的键，在std::unordered_map中，键都会通过内部的哈希函数转换为哈希值。然后，这个哈希值被用来在哈希表的内部结构中定位和访问对应的值

最后关于解决哈希冲突，开放寻址法（Open Addressing）和链表法（Chaining）是解决哈希表中哈希冲突的两种常见方法。它们的主要区别在于处理冲突的方式不同。

开放寻址法：

  • 当插入的元素发生冲突时，即该位置已被占用，开放寻址法会寻找哈希表中的下一个空闲位置。
  • 实现方式包括线性探测、二次探测和双重散列等。
  • 开放寻址法的优点是实现简单，数据存储在一个连续的数组中，可以利用CPU缓存加快访问速度。
  • 缺点是删除操作复杂，容易产生聚集问题，导致性能下降。

链表法：

  • 每个哈希表的槽位对应一个链表。所有散列值相同的元素都会被添加到相同槽位对应的链表中。
  • 当发生冲突时，元素会被直接添加到链表的末尾。
  • 链表法的优点是处理冲突简单，扩展容易，不会产生聚集问题。
  • 缺点是链表的指针需要额外的内存空间，且链表的遍历可能会因为CPU缓存不命中而较慢。

NULL

在C++中，NULL和nullptr都用于表示空指针，但它们之间存在一些重要的区别：

  • NULL是一个宏，通常被定义为整数0。在C++中，NULL可能会导致类型歧义，因为它可以被视为整数而不仅仅是指针。
  • nullptr是C++11中引入的一个关键字，它的类型是std::nullptr_t。nullptr专门用于表示空指针，它可以隐式转换为任何指针类型，但不会转换为整数类型，这避免了与整数的歧义。

例如，如果你有两个重载的函数，一个接受整数类型参数，另一个接受指针类型参数，使用NULL可能会导致调用整数类型的版本，而使用nullptr则会明确调用指针类型的版本。这就是nullptr相较于NULL的一个优势。

此外，nullptr不仅清晰地表达了程序员的意图，即表示空指针，而且还提供了更好的类型安全性。

额外内容：

  1. Entity() = default;

在C++中，Entity()=default;是C++11标准中引入的一种语法，用于告诉编译器使用默认生成的构造函数。这意味着你不需要为这个构造函数编写任何特定的实现代码，编译器会为你生成一个标准的构造函数。

这个功能特别有用，因为在某些情况下，如果你已经提供了其他构造函数，编译器不会自动生成默认构造函数。使用=default可以确保即使在这些情况下，你也有一个默认的构造函数。

此外，使用=default可以保持类的某些属性，比如让类保持为聚合类型或者平凡类型。如果你想让你的类保持为聚合类型或者平凡类型，你就需要使用=default，而不是提供一个空的构造函数实现12。

简而言之，Entity()=default;的作用是：

  • 告诉编译器生成默认的构造函数。
  • 确保即使定义了其他构造函数，类也有一个默认构造函数。
  • 保持类的聚合或平凡类型的特性。
