在 epoll 事件通知机制中，事件类型如 EPOLLIN、EPOLLOUT 等被定义为二进制中的单个位标志（bit flags）。这些标志是通过位掩码（bitmask）来表示的，而不是通过简单的数字相加。每种类型的事件都对应于整数值中的一个特定的位位置，而这些位位置是通过位运算来设置和检查的。

例如，EPOLLIN 可能被定义为 0x001（二进制的 0001），而 EPOLLOUT 可能被定义为 0x004（二进制的 0100）。当一个文件描述符同时有可读和可写事件发生时，epoll 会将这两个事件的位掩码进行按位或运算（|），而不是数字相加。这样，即使两个事件同时发生，EPOLLIN 的位也会保持为 1。

这就是为什么我们可以通过检查按位与运算的结果是否为非零来确定特定事件（如 EPOLLIN）是否发生，而不用担心其他事件的影响。

正是通过位掩码（bitmask）的设计，我们可以确保即使多个事件同时发生，也能通过按位与运算（&）得到正确的结果。位掩码是一种使用二进制位操作来表示和控制多个配置选项的技术。在 epoll 事件中，每个事件类型都有一个唯一的位掩码，这些掩码在设计时就确保了它们之间的二进制表示不会冲突。

例如，EPOLLIN 和 EPOLLOUT 可能分别被定义为：

  • EPOLLIN: 0x001 或 0001（二进制）
  • EPOLLOUT: 0x004 或 0100（二进制）

这种设计允许我们使用单个整数值来存储和检查多个事件的发生情况，而不需要为每个事件单独使用一个变量。这样做不仅节省了空间，而且提高了效率，因为位运算通常比其他类型的运算更快。

条件变量

条件变量condition variable 是C++11引入的一种同步原语，用于实现线程之间的等待和唤醒机制。它通常与互斥锁（mutex）结合使用，以实现复杂的线程同步和通信。

  1. 释放锁以避免死锁

    当一个线程调用wait时，它会进入等待状态。如果在等待期间不释放互斥锁，其他线程将无法获取锁，从而无法改变条件变量的状态，这会导致死锁。因此，wait函数在进入等待状态前会释放互斥锁。

  2. 重新获取锁以确保线程安全

    当条件变量被唤醒时，wait函数会重新获取互斥锁，以确保在检查条件变量状态时，线程对共享资源的访问是安全的。

  3. 使用std::unique_lock的灵活性

    std::unique_lock提供了比std::lock_guard更灵活的锁管理功能。它允许在锁定和解锁之间进行更多的控制，例如延迟加锁、提前解锁等。这种灵活性对于条件变量的使用非常重要，因为wait函数需要在内部释放和重新获取锁。

mutex

一个同步原语，用于保护共享数据不被多个线程同时访问。它提供了独占的、非递归的所有权语义，确保同一时刻只有一个线程可以访问被保护的数据。

是用于保护共享数据免受多个线程同时访问的同步原语。互斥量提供独占的、非递归的所有权语义：调用线程从成功调用lock或try_lock开始拥有互斥量，直到调用unlock为止

  1. 互斥锁的操作对象：lock和unlock操作的是互斥锁本身（即mtx），这些操作改变了互斥锁的状态（锁定或解锁）。

互斥锁的工作原理

  • 加锁（lock）：当一个线程调用lock()时，如果互斥锁是未锁定状态，线程将成功获取锁并进入临界区；如果互斥锁是锁定状态，线程将被阻塞，直到锁被释放。
  • 解锁（unlock）：当一个线程调用unlock()时，它会释放互斥锁，使其他被阻塞的线程有机会获取锁并进入临界区。

也就是当前线程拥有这个锁后，就可以使用这个锁对选定的部分操作进行上锁操作。

IO多路复用

IO多路复用（Input/Output Multiplexing）是一种在单个线程中管理多个输入/输出通道的技术2。它允许一个线程同时监听多个输入流（例如网络套接字、文件描述符等），并在有数据可读或可写时进行相应的处理，而不需要为每个通道创建一个独立的线程。常见的IO多路复用机制包括select、poll和epoll，这些机制通过将多个IO通道注册到一个事件管理器中，然后通过阻塞方式等待事件的发生

IO多路复用的工作原理是通过一种机制，允许单个进程监视多个文件描述符，一旦某个文件描述符就绪（例如，一个网络连接上有数据可读），就能够通知程序进行相应的读写操作。

这里的“多路”指的是多个网络连接或文件流，“复用”指的是使用同一个进程或线程来处理多个流。

在传统的阻塞IO模型中，每个IO操作通常需要一个独立的线程来处理，这在处理大量并发连接时会导致线程数量的增加，从而带来线程切换和上下文切换的开销。而IO多路复用通过使用一个线程来监听多个IO事件，避免了线程数量的增加，减少了这些开销

常见的IO多路复用技术包括select、poll和epoll

  • select和poll：这两种机制都是通过告诉内核监视一系列的文件描述符，然后进程阻塞并等待内核通知它哪些文件描述符就绪了。select有最大文件描述符数量的限制，而poll没有这个限制。但它们都有一个共同的缺点，那就是当监视的文件描述符数量很多时，每次调用都需要遍历整个集合来检查状态，效率较低1。

  • epoll：与select和poll不同，epoll是在Linux 2.6及以上版本中提供的一种高效的IO事件通知机制。它不需要遍历整个文件描述符集合，而是当文件描述符状态发生变化时，就会将这个事件通知给应用程序。epoll可以处理数以万计的并发连接，而且效率不会随着文件描述符数量的增加而降低。epoll的另一个特点是它支持“边缘触发”和“水平触发”，前者只在状态变化时通知一次，而后者只要条件满足就会不断通知1

虽然epoll在处理大量并发连接时比select和poll更高效，但这并不意味着select和poll就完全没有用处了。select和poll在某些情况下仍然有其适用性和优势。

select&poll

select与poll的逻辑相似，都是把要监听的socket存入一个数据结构中（select-bitmap）然后每次发生事件后循环遍历这个数据结构返回发生事件的socket，这里的事件可能是读事件/写事件。都是水平触发( level-trigger)

读事件：1）已连接队列中有已经准备好的socket（有新的客户端连上来了）；
2）接收缓存中有数据可以读（对端发送的报文已到达）；
3）tcp连接已断开（对端调用close()函数关闭了连接）。
写事件：发送缓冲区没有满，可以写入数据（可以向对端发送报文）。

select

select 的实现思路很直接，假如程序同时监视sock1、sock2 和 sock3 三个socket，那么在调用select 之后，操作系统把进程分别加入这三个socket的等待队列中. 每次调用select都需要将进程加入到所有监视socket的等待队列，每次唤醒都需要从每个队列中移除。这里涉及了两次遍历，而且每次都要将整个 FDS 列表传递给内核，有一定的开销.正是因为遍历操作开销大，出于效率的考量，才会规定select 的最大监视数量，默认只能监视1024个socket。

epoll

epoll是在select 出现 N 多年后才被发明的，是select 和 poll的增强版本。epoll通过以下一些措施来改进效率：

1.功能分离

​ select 低效的原因之一是将“维护等待队列”和“阻塞进程”两个步骤合二为一。

epoll将这两个操作分开，先用epoll_ctl 维护等待队列，再调用epoll_wait 阻塞进程。显而易见，效率就能得到提升。

2.就绪列表

select低效的另一个原因在于程序不知道哪些socket收到数据，只能一个个遍历。如果内核维护一个“就绪列表”，引用收到数据的socket，就能避免遍历。

实现

当某个进程调用epoll_create方法时，内核会创建一个 eventpoll 对象（也就是程序中 epfd 所代表的对象）。eventpoll 对象也是文件系统中的一员，和socket一样，它也会有等待队列。

创建一个代表该epoll的 eventpoll 对象是必须的，因为内核要维护“就绪列表”等数据，“就绪列表”可以作为 eventpoll 的成员。epoll使用双向链表来实现就绪队列，使用红黑树作为索引结构

创建epoll对象后，可以用epoll_ctl 添加或删除所要监听的socket。以添加socket为例。

当socket收到数据后，中断程序会操作 eventpoll 对象，而不是直接操作进程。

当socket收到数据后，中断程序会给 eventpoll 的“就绪列表”添加socket引用。

eventpoll 对象相当于socket和进程之间的中介，socket的数据接收并不直接影响进程，而是通过改变 eventpoll 的就绪列表来改变进程状态。

当程序执行到epoll_wait 时，如果rdlist 已经引用了socket，那么epoll_wait 直接返回，如果 rdlist 为空，阻塞进程。

在实际的使用中，IO多路复用要求不能阻塞在任意环节，所以应该设置非阻塞

整体逻辑：将所有需要epoll监控的socket通过epoll_event（这个结构体更是为了设置socket的一个临时方法，并不存储实际事件，只用来保存当前要进行设置的fd）来设置其需要监视的事件类型（读/写）以及触发方式（边缘触发/水平触发），然后通过epoll_ctl指定EPOLL_CTL_ADD来加入到epoll的fd中，然后就可以用过epoll_wait开始监视事件，并返回发生事件的socket到第epoll_wait的二个参数中，处理事件后使用epoll_ctl指定EPOLL_CTL_DEL删除对应的socket。

实现一个不可继承的类

过度使用继承可能会导致一些问题，这就是为什么现代编程实践中建议“少用继承，多用组合”的原因。

使用组合而不是继承。在C++中，如果你想要防止一个类被继承，可以使用以下几种方法：

  1. 私有化构造函数：通过将构造函数设置为私有，其他类就无法继承它，因为它们无法调用基类的构造函数。这种方法通常与一个静态成员函数配合使用，该函数负责创建类的实例1。

私有化构造函数之后就会使得在类外部都无法创建实例，虽然保证了类无法被继承，但是导致了类本身也无法创建实例，解决这个问题考虑：

​ a. 静态成员函数：可以在类内部提供一个静态成员函数来创建对象。这个函数可以访问类的私有成员，包括私有构造函数。这种方法常用于实现单例模式1。
​ b. 友元类或友元函数：可以声明一个友元类或友元函数，它们可以访问类的私有成员，包括私有构造函数。这样，友元可以负责创建类的实例2。
​ c. 工厂模式：可以使用工厂模式来创建对象。在这种模式下，一个专门的工厂类负责对象的创建，它可以访问原始类的私有构造函数来创建对象3。

  1. 虚继承和私有基类构造函数：使用模板和虚继承来阻止类被继承。这种方法涉及到一个基类的构造函数是私有的，而派生类是该基类的虚继承者1。

  2. 使用final关键字：从C++11开始，可以在类声明的末尾使用final关键字来防止类被继承。这是一种简单直接的方法

信号与槽

一旦信号和槽通过 connect 函数连接起来，每当信号被 emit 发射，它就会触发槽函数的执行。这是Qt事件驱动编程的核心机制，允许对象之间以松耦合的方式进行通信。

槽函数的调用是由emit显式发送信号函数来触发的。

自定义信号:

  • 在类的声明中，使用 signals: 关键字来定义信号。
  • 信号没有返回值，可以有参数。
  • 信号是函数的声明，不需要定义实现。
  • 使用 emit 关键字来发射信号。

自定义槽:

  • 在类的声明中，使用 slots: 或者 public/private/protected slots: 来定义槽函数。
  • 槽函数的实现与普通成员函数相同。
  • 槽函数通常没有返回值，参数需要与信号匹配。

连接信号和槽:

  • 使用 QObject::connect 函数来连接信号和槽。
  • connect 的参数包括信号发送者、信号、信号接收者和槽函数。

发射信号:

  • 使用 emit 关键字来发射信号，触发连接的槽函数。

类必须包含 Q_OBJECT 宏，这样 Qt 的元对象系统才能正常工作，支持信号和槽的功能

why?

  1. 为什么信号函数不允许有返回值？ 信号函数不允许有返回值是因为它们的目的是通知其他对象某个事件发生了，而不是进行计算并返回一个值。信号的设计是为了触发槽函数的执行，而槽函数可以有返回值。信号的作用是通知，而不是询问。
  2. 信号函数的参数有什么作用？ 信号函数的参数用于传递与信号相关的数据。例如，如果你有一个信号表示温度变化，参数可能就是新的温度值。这样，当信号被发射时，槽函数可以接收这个值并据此做出反应
  3. 没有参数的信号如何工作？ 没有参数的信号简单地通知接收者某个事件发生了，而不提供额外的数据。例如，一个按钮被点击的信号可能不需要参数，因为接收者只需要知道按钮被点击了，而不需要其他信息。
  4. 使用emit发射信号是什么意思？ emit是Qt中的一个关键字，用于发射信号。当你想通知其他对象某个事件发生时，你会在你的代码中使用emit来发射相应的信号。这样，所有连接到该信号的槽函数都会被调用
