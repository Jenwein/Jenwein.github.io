模板

在C++和其他一些编程语言中，模板是一种通用编程的工具，允许程序员编写通用代码，以适应不同的数据类型或数据结构。模板的目的是提供一种在编写代码时延迟数据类型或数值选择的方式。

在C++中，主要有两种模板：函数模板和类模板。

  1. 函数模板：允许定义通用的函数，其中某些类型或数值可以通过模板参数进行参数化。函数模板的声明以 template 关键字开始，后面跟着模板参数列表和函数的定义。

  2. 类模板：允许定义通用的类，其中某些类型或数值可以通过模板参数进行参数化。类模板的声明也以 template 关键字开始，后面跟着模板参数列表和类的定义。

通过使用模板，程序员可以编写更通用、灵活且可重用的代码，因为模板允许在编译时期实现参数化。在实际使用中，模板常常与泛型编程概念相结合，以实现更灵活的数据结构和算法。

泛型编程是一种编程范式，它允许程序员编写与数据类型无关的代码。这是通过使用模板来实现的，模板可以看作是对类型进行参数化的工具。泛型编程的一个主要优势是代码重用：你可以创建能够处理多种数据类型的函数和类，而不需要为每种类型编写特定的代码。如果你想创建一个能够对任意类型的数组进行排序的函数，你可以使用函数模板来实现

  1. 模板参数推导：这是编译器自动确定模板参数类型的过程。当你使用模板函数或类时，编译器可以根据传递给模板的实参来推导模板参数的类型。
  2. 可变参数模板：C++11引入了可变参数模板，允许模板接受任意数量的模板参数。这使得编写能够接受不同数量参数的函数和类成为可能。
  3. 模板的默认参数：你可以为模板参数提供默认类型，如果在模板实例化时没有指定参数类型，将使用默认类型。
  4. 非类型模板参数：模板不仅可以接受类型作为参数，还可以接受常量值作为参数，这些值必须在编译时已知。
  5. 模板模板参数：这是一种高级的模板编程技术，允许模板接受另一个模板作为参数。
  6. 模板别名：C++11还引入了模板别名，它允许你为复杂的模板类型提供一个简短的别名。
  7. SFINAE（Substitution Failure Is Not An Error）：这是一种编译时机制，用于在模板参数替换失败时选择合适的函数重载，而不是产生编译错误。
  8. 模板类的成员模板：一个模板类可以拥有模板成员函数，这些成员函数本身也是模板。
  9. 概念和约束（C++20）：这是C++20中引入的新特性，允许你为模板参数指定必须满足的条件，从而提供更清晰的模板接口和更好的错误信息。

.lib和.dll

lib是和dll对应的。lib是静态链接库的库文件，dll是动态链接库的库文件。
所谓静态就是link的时候把里面需要的东西抽取出来安排到你的exe文件中，以后运行你的exe的时候不再需要lib。
所谓动态就是exe运行的时候依赖于dll里面提供的功能，没有这个dll，你的exe无法运行。

.lib

lib文件是一种库文件，它可以分为两种类型：静态库和导入库。

静态库包含了函数的实际代码，可以在编译时直接链接到可执行文件中，不需要依赖其他文件。

导入库只包含了函数在动态链接库（dll）中的地址信息，可以在运行时动态加载dll文件，实现函数的调用。

lib库会做链接过程，但具体的方式取决于它的类型。如果是静态库，那么它会在编译时被链接器合并到可执行文件中，形成一个完整的程序。如果是导入库，那么它会在运行时被操作系统加载到内存中，然后通过地址映射来调用dll中的函数。

.dll

DLL导出函数，可供应用程序调用；
DLL内部函数，只能在DLL程序使用，应用程序无法调用它们

DLL文件作用是: 函数可执行代码

.lib&.dll

C++创建一个动态链接库，编译后会生成两个可用的文件一个是lib文件一个是dll文件，那么这个lib文件是干嘛的呢？
在使用动态库的时候，往往提供两个文件：一个引入库和一个DLL。引入库包含被DLL导出的函数和变量的符号名，DLL包含实际的函数和数据。在编译链接可执行文件时，只需要链接引入库，DLL中的函数代码和数据并不复制到可执行文件中，在运行的时候，再去加载DLL，访问DLL中导出的函数。

  1. Load-time Dynamic Linking 载入时动态链接
    这种用法的前提是在编译之前已经明确知道要调用DLL中的哪几个函数，编译时在目标文件中只保留必要的链接信息，而不含DLL函数的代码；当程序执行时，利用链接信息加载DLL函数代码并在内存中将其链接入调用程序的执行空间中，其主要目的是便于代码共享。
  2. Run-time Dynamic Linking 运行时动态链接
    这种方式是指在编译之前并不知道将会调用哪些DLL函数，完全是在运行过程中根据需要决定应调用哪个函数，并用LoadLibrary和GetProcAddress动态获得DLL函数的入口地址。

lib文件和dll文件的关系如下：

如果一个程序使用了静态库，那么它不需要dll文件，因为它已经包含了所有的函数代码。
如果一个程序使用了导入库，那么它需要dll文件，因为它只有函数的地址信息，需要在运行时加载dll文件才能找到函数的实际代码。
如果一个程序直接使用了dll文件，那么它不需要lib文件，因为它可以通过API函数（如LoadLibrary和GetProcAddress）来获取dll文件中的函数地址

  • dll.lib实际上是一种静态导入库，与这个dll一起使用，包含了.dll的所有的函数，符号的位置，所以可以在编译时链接他们，如果没有这个文件则需要通过函数名来访问dll文件内的函数

补充

vector中的reserve和resize的区别是：

  • reserve是预留内存空间，它只改变vector的容量（capacity），不改变vector的大小（size）。它可以避免多次不必要的扩容和内存拷贝，提高效率。

  • resize是改变vector的大小，它会增加或删除vector的末尾的元素，从而改变vector的容量和大小。它可以直接访问vector中的元素，而不需要调用push_back或insert

vector分配策略

刚开始了解时有个疑问，vector是动态数组，不应该有多少元素就有多大的容量吗，为什么会有空闲呢？

​ vector是动态数组，但是它也有一些内存分配和管理的策略，为了提高效率和性能，避免频繁的内存重新分配和元素拷贝。vector会根据不同的情况，预留一些额外的空间，用于存放新添加的元素，这样就不必为每个新元素重新分配整个容器的内存空间。这些额外的空间就是vector的空闲容量（capacity - size）

这里涉及到一些其他的小东西：

  • vector的扩容机制不一定是每次加1，而是可能按照一定的倍数增长，比如1.5倍或者2倍123。这样可以减少扩容的次数，提高效率。
  • vector的扩容机制也不一定是每次拷贝所有的元素，而是可能使用移动构造函数，把旧的元素移动到新的内存空间45。这样可以避免不必要的拷贝开销，尤其是对于一些不支持拷贝的类型，比如unique_ptr。
  • vector的扩容机制也不一定是每次重新分配一块新的内存，而是可能使用realloc函数，尝试在原来的内存空间后面扩展，如果成功就不需要拷贝或移动元素6。这样可以节省内存和时间，但是这种方法只适用于一些可平凡复制的类型，比如int或char。

内存对齐

一种编程技术，用于让数据在内存中的存放位置满足一定的规则，以提高数据访问的效率和兼容性。填充字节是为了实现内存对齐而在数据之间或结构体末尾添加的额外字节，用于占位或对齐。

内存对齐的原因有两个：

  • 平台原因：不同的硬件平台对数据的访问方式和要求可能不同，有些平台只能从特定的地址开始访问特定类型的数据，否则会出现错误或异常。
  • 性能原因：内存对齐可以使数据访问更快，因为处理器可以一次读取或写入一个数据单元，而不需要多次操作或额外的处理。

内存对齐的规则有三个：

  • 基本类型的对齐值就是其大小，例如 int 的对齐值是 4，char 的对齐值是 1。
  • 结构体或联合体的成员，第一个成员的偏移量是 0，以后每个成员的偏移量都是该成员的对齐值和预设的对齐系数中较小的那个的整数倍，如果需要，编译器会在成员之间插入填充字节。
  • 结构体或联合体本身的对齐值是其最大成员的对齐值和预设的对齐系数中较小的那个，结构体或联合体的总大小是其对齐值的整数倍，如果需要，编译器会在结构体或联合体末尾添加填充字节。

预设的对齐系数是编译器默认的内存对齐参数，通常是 4 或 8，也可以通过编译指令或选项来修改。例如，在 gcc 中，可以使用 #pragma pack(n) 来指定对齐系数为 n，n 可以是 1，2，4，8，16 等。

什么是浅拷贝呢

​ 浅拷贝是创建一个新对象，其属性值是原始对象属性值的一份精确拷贝。如果属性是基本类型，拷贝的就是基本类型的值；如果属性是引用类型，拷贝的就是内存地址，因此如果其中一个对象改变了这个地址，就会影响到另一个对象。简单来说，浅拷贝只拷贝对象不拷贝引用

​ 如果你有一个包含引用类型成员的对象，并且你对这个对象进行浅拷贝，那么拷贝出来的对象中的引用类型成员仍然指向原始对象中的相同实例。这意味着，如果你修改了原始对象中的引用类型成员，拷贝出来的对象中的相应成员也会受到影响，因为它们指向的是同一个实例。

操作符”->”

->是一个成员访问的操作，它的作用是通过一个指针来访问它所指向的对象的成员变量或成员函数。它的左边必须是一个指针，右边必须是一个成员的名字。它的含义是，先解引用左边的指针，得到它所指向的对象，然后访问这个对象的右边的成员。也没什么。

“.”和”->”的区别

创建个类先

对象要调用Print方法就是e.Print(); 或ptr->Print(); 这两个对象一个是Entity，一个是Entity* 。

在C++中，. 和 -> 都是用来访问结构体或类的成员的运算符，但它们的使用场景不同。

  • . 运算符用于访问一个实际对象的成员。例如，如果你有一个 Vector3 类型的对象 vec，你可以使用 vec.x 来访问它的 x 成员。
  • -> 运算符用于通过指针访问对象的成员。如果你有一个指向 Vector3 对象的指针 ptr，你可以使用 ptr->x 来访问它指向的对象的 x 成员。

简单来说，当你有一个对象时，你使用 . 来访问它的成员；当你有一个指向对象的指针时，你使用 -> 来访问它指向的对象的成员。这是因为指针需要被解引用（dereferenced）来访问指向的对象，而 -> 运算符自动为你做了这个操作。

SmartPtr(智能指针)

  • 我们都知道new和delete，new在堆上分配内存，而delete用来删除对象和释放内存，因为new分配的内存不会自动释放，智能指针是自动实现这个操作的一种方式，也就是说调用new分配内存时，不用再去调用delete，甚至不用调用new。

  • 当你创建一个智能指针，它会调用new为你分配内存，然后基于你使用的智能指针（unique和shared），分配的内存会在某一时刻自动释放

  • 使用时需要包含头文件#include <memory>

接下来详细讲解几个智能指针，先写个类再说

unique_ptr(唯一指针)

当指针超出作用域时就会被销毁，就会调用delete，但是正如它的名字，unique，这个指针是不可以复制的，如果你copy了一个unique_ptr，那么它们两个指向相同的内存，当一个指针die了，这块内存就会被释放，另一个unique_ptr就指向了已经被释放的内存

聚个离子：

有了刚才的那个Entity类，我们就可以在main中定义unique_ptr，

std::unique_ptr<Entity>entity(new Entity());

如果在 new Entity() 表达式执行过程中发生异常，那么为新对象分配的内存将不会被释放，因为 unique_ptr 的构造函数还没有开始执行，也就没有机会接管这块内存。

所以出于异常安全考虑写为

std::unique_ptr<Entity>entity=std::make_unique<Entity>();

std::make_unique 会在内部创建一个新对象，并且立即将它包装在一个 unique_ptr 中。如果在对象的创建过程中发生异常，unique_ptr 会负责清理已经分配的内存，因此不会发生内存泄漏。

这样我们就有了一个entity，既分配在堆上，又能自动new和delete，在作用域结束时就会自动delete掉。

但是不能copy，例如如果再写一个std::unique_ptr<Entity>e=std::make_unique<Entity>(); 然后e=entity； 就会报错

是一种保护措施，通常尽可能使用unique_ptr，因为它的开销较低，但是如果你需要在对象之间共享，就不能使用unique_ptr，而是应该使用shared_ptr

使用注意

  • 不要用同一个裸指针初始化多个 unique_ptr 对象

正确的做法是为每个 unique_ptr 分配一个新的内存区域，或者使用 std::move 来转移所有权，确保任何时候只有一个 unique_ptr 拥有特定内存的所有权。这样可以避免多次删除同一内存块的问题。

  • get() 方法用于获取智能指针内部的裸指针（raw pointer）也就是原始指针。这个方法不会改变智能指针的所有权，只是提供了一种访问其管理的资源的方式

使用 get() 方法时要小心，因为如果 unique_ptr 被销毁或者其所有权被转移，那么通过 get() 获得的裸指针将指向无效的内存。因此，确保在 unique_ptr 存在的生命周期内使用裸指针，避免悬挂指针（dangling pointer）的问题。

  • 不要用unique_ptr管理不是new分配的内存

因为 unique_ptr 默认使用 delete 来释放它所拥有的内存。如果你用 unique_ptr 管理了一个不是通过 new 分配的内存块，当 unique_ptr 试图释放这块内存时，会使用 delete，这可能导致未定义行为，因为只有 new 分配的内存才应该用 delete 来释放。

在这个例子中，a 是在栈上分配的，而不是通过 new。当 ptr 离开作用域时，它会尝试用 delete 来释放 a，但这是不正确的，因为 a 不是动态分配的内存。

std::unique_ptr<_Ty[],typename _Dx> 第一个参数为指针类型，第二个参数指定一个删除器，删除器（Deleter）是一个函数或者函数对象，它被用来自定义智能指针如何释放其所管理的资源。当智能指针如 unique_ptr 或 shared_ptr 被销毁时，删除器会被调用来执行资源的清理工作。

通常，智能指针会使用默认的删除器，即对于 unique_ptr 是 delete，对于 shared_ptr 是 delete 或 delete[]，这适用于通过 new 或 new[] 分配的动态内存。但是，如果你管理的资源需要特殊的清理过程，比如通过 malloc 分配的内存需要用 free 来释放，或者你需要关闭一个文件句柄或网络套接字，那么你就需要提供一个自定义的删除器。

这个例子中，我们定义了一个名为 closeFile 的函数，它接受一个 FILE* 类型的参数，并使用 fclose 来关闭文件。然后，我们创建了一个 unique_ptr，它的第二个模板参数是一个函数指针类型 void(*)(FILE*)，这表明这个 unique_ptr 将使用一个函数指针作为其删除器。在 unique_ptr 的构造函数中，我们使用 fopen 来打开一个文件，并将 closeFile 作为删除器传递给 unique_ptr。

这样，当 unique_ptr 离开作用域时，closeFile 会被自动调用来关闭文件，确保资源被正确管理和释放。

  • 用于函数的参数时，由于时unique指针，无法拷贝，所以只能进行引用传递和指针传递（配合get()方法）

shared_ptr(共享指针)

shared_ptr的工作方式是通过引用计数，引用计数是一种跟踪统计你的指针有多少引用的方法，一旦计数为零，它就会被删除。类似的，定义方式 std::shared_ptr<Entity>SharedEntity = std::make_shared<Entity>();

当然也可以是 std::shared_ptr<Entity>SharedEntity(new Entity());但是这里不这样不是异常安全问题，是因为shared_ptr会额外分配一块叫做控制块的内存，用来存储引用计数如果你先创建一个new Entiry，然后再把它传给shared_prt的构造函数，就会有两次内存分配，一个new Entity 一个shared_ptr的控制块的分配.

梨子梨子：

先是简单的创建一个SharedEntity，所以,引用计数+1=1，然后再这个作用域外再写一个作用域并创建e0，然后e0 = SharedEntity;，这个对象的引用+1=2，在第十行虽然shared_ptr被销毁但是e0未结束，所以计数为-1=1，在第12行e0结束，计数归零，所以打印”Destroyed Entity!”，释放内存

weak_ptr

这里还有一个weak_ptr,可以与shared_ptr一起使用，与shared_ptr不同的是不会增加引用计数

cherno：“因此，如果您不想获得Entity的所有权，则这非常有用，就像你可能在对一个Entity列表进行排序，你并不真正关心它们是否有效，你只需要存储一个它们的引用就好了，你可以通过weak ptr去判断底层对象是否还存活”
