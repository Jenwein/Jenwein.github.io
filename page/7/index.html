文件操作

C++ 的文件流（fstream）以及相关的文件操作是用于在 C++ 中进行文件输入输出的机制。它们提供了一种方便的方式来读取和写入文件内容，包括文本文件和二进制文件。让我详细讲解一下：

文件流（fstream）类：

在 C++ 中，文件流（fstream）类是用于文件输入输出的类，它继承自基类 iostream。有三个主要的文件流类：

  1. ifstream：用于从文件中读取数据，通常用于读取文本文件或者二进制文件。
  2. ofstream：用于向文件中写入数据，通常用于创建新文件或者覆盖已有文件的内容。
  3. fstream：用于既可以读取也可以写入文件的操作，是 ifstream 和 ofstream 的结合。

这些类都定义在 <fstream> 头文件中。

文件描述符

文件描述符（File Descriptor，简称FD）是操作系统中用于访问文件和I/O资源的抽象表示。在Unix和类Unix系统中，文件描述符是一个整数，它唯一地标识了进程已打开的文件、管道、网络连接等资源。这里的“文件”不仅指传统意义上的文件，还包括任何可进行I/O操作的资源。

文件描述符的基本概念

  • 整数索引：文件描述符是一个非负整数。当打开一个文件时，系统会返回一个文件描述符，用于后续的读写操作。
  • 标准输入输出：程序启动时，默认会打开三个文件描述符：0（标准输入stdin）、1（标准输出stdout）、2（标准错误stderr）。
  • 资源引用：文件描述符提供了一个统一的接口来引用各种类型的I/O资源，如普通文件、目录、套接字等。

文件描述符的工作原理

当一个进程打开或创建文件时，内核会创建一个文件描述符，并将其返回给进程。这个描述符实际上是一个索引，指向内核中的一个文件表项。这个表项包含了文件的各种信息，如当前的读写位置、文件的状态和访问权限等。

文件描述符的存储和管理

  • 进程文件描述符表：每个进程都有一个文件描述符表，用于跟踪该进程打开的所有文件描述符。
  • 系统级文件表：系统维护一个全局的文件表，记录所有打开的文件的状态和属性。
  • inode表：文件系统维护一个inode表，存储文件的元数据，如文件大小、权限、创建和修改时间等。

文件描述符的使用场景

  • 文件操作：通过文件描述符，进程可以对文件执行读写、定位、锁定等操作。
  • 网络通信：套接字也会被分配文件描述符，进程可以通过它来进行网络数据的发送和接收。
  • 进程间通信：管道和FIFO等IPC机制也使用文件描述符来传递数据。

文件描述符的限制

  • 数量限制：每个进程能打开的文件描述符数量是有限的，这个限制可以通过系统调用或配置文件来调整。
  • 生命周期：文件描述符在进程生命周期内有效，当进程终止时，它打开的所有文件描述符都会被自动关闭。

文件描述符是Linux/Unix系统编程中的一个核心概念，理解它对于进行系统级编程至关重要。它提供了一个简洁的抽象，使得对各种类型的I/O资源的操作变得统一和简单

流(stream)

在 C++ 中，流（stream）是一个抽象概念，表示数据的流动或者流向。流可以是输入流（input stream）或输出流（output stream），用于从某个源头读取数据或者向某个目的地写入数据。

流的概念源自于 I/O 设备（如键盘、屏幕、文件等）之间的数据传输过程。流将数据抽象为一系列的字符或字节，通过特定的方式从源头流向目的地。流可以是单向的（只能进行输入或输出）或者双向的（既可以进行输入也可以进行输出）。

在 C++ 中，流被实现为一系列的类，分别用于处理不同类型的数据流。常见的流类包括：

  1. 输入流类（input stream classes）：用于从源头读取数据，例如键盘、文件等。常见的输入流类有 std::istream、std::ifstream 等。
  2. 输出流类（output stream classes）：用于向目的地写入数据，例如屏幕、文件等。常见的输出流类有 std::ostream、std::ofstream 等。
  3. 文件流类（file stream classes）：用于文件的输入输出操作。常见的文件流类有 std::ifstream、std::ofstream、std::fstream 等。

这些流类提供了一系列的成员函数和操作符，用于从流中读取数据或者向流中写入数据。通过使用这些流类，你可以方便地进行文件操作、标准输入输出、字符串流等各种输入输出操作。

总的来说，流是 C++ 中用于处理输入输出的抽象概念，通过流类来实现对数据的读取和写入，使得输入输出操作更加方便、灵活、可扩展。

2024-05-16 补 输出缓冲

流，字如其名，像是水流，cout，cerr以及自定义的流file这些看作水流本身，流式输入输出由运算符<</>>来指定水流方向。

创建一个流式对象，例如输出文件流对象ofstream outFile ，对象outFile本身就是一个水流，当其与任意文件关联，也就是outFile.open("filename")，之后的操作都是对关联的文件的操作，outFile << "Hello, file!"; 像是将字符串放在水流中outFile中，因为outFile关联了文件filename，所以水流的终点就是这个文件内部。

水流并不是畅行无阻的，有时候中间会有水坝的存在(缓冲机制)，正常情况下，水（数据）会在水坝（缓冲区）中积累，直到达到一定的量，然后水坝会放水（刷新缓冲区），将积累的水（数据）一次性地流向下游（屏幕或文件）。

IO操作是消耗性能的，缓冲机制就是减少IO的次数，它可以减少对下游（屏幕或文件）的写入操作次数，从而提高效率。如果没有缓冲区，每次向 std::cout 添加数据都会立即写入屏幕，这会导致大量的I/O操作，降低程序的性能。

C++中存在流操纵器用于控制输入/出流，一个常见的 unitbuf 将导致输出在每次插入后进行刷新。也就是数据生成即立即输出，不会被缓存，使用 std::unitbuf 会启用 unitbuf 标志，如同调用 str.setf(std::ios_base::unitbuf)。相反，使用 std::nounitbuf 会禁用 unitbuf 标志，如同调用 str.unsetf(std::ios_base::unitbuf)

使用：std::cout << std::unitbuf; // 启用自动刷新

手动打开水坝QAQ，例如，如果你正在写一个需要实时日志记录的程序，你可能会希望日志信息能够立即显示在屏幕上或写入文件，而不是在程序崩溃或异常退出时可能丢失未刷新的数据

在这个程序中，我们有三种不同的输出方式：

  1. 第一种情况，我们只是简单地使用 std::cout 输出文本，没有立即刷新。你会注意到，“开始计算…” 和 “计算完成。” 几乎同时出现，因为在耗时操作期间，输出被缓存了。
  2. 第二种情况，我们在输出 “开始计算…” 后立即使用 std::flush 刷新缓冲区。这会导致 “开始计算…” 立即出现，然后程序会等待3秒，最后输出 “计算完成。”。
  3. 第三种情况，我们使用 std::endl 来结束输出。std::endl 会插入一个换行符并刷新缓冲区，所以 “开始计算…” 会立即出现，然后程序等待3秒，最后输出 “计算完成。”。

标准输出流cout就有缓冲机制而标准错误流cerr则是立即刷新，缓冲区刷新还有其他情况：

  • 缓冲区满时。
  • 程序正常结束时，标准库会清理所有的I/O流，包括刷新 std::cout 的缓冲区。
  • 当你输出一个换行符（'\n'）时，如果 std::cout 被设置为行缓冲模式，它会刷新缓冲区。这是很多环境的默认行为，尤其是当输出到终端时。
  • 使用流操纵符 std::endl，它会插入一个换行符并刷新缓冲区。
  • 显式调用 std::flush，它会刷新缓冲区而不输出任何字符

输出流对象是字节的目标。 三个最重要的输出流类为 ostream、ofstream 和 ostringstream。

ostream 类通过派生类 basic_ostream 支持预定义的流对象：

  • cout 标准输出
  • 具有有限缓冲的 cerr 标准错误
  • clog 类似于 cerr，但具有完全缓冲

很少从 ostream 构造对象；通常使用预定义的对象。 在某些情况下，可以在启动程序后重新分配预定义的对象。 可配置为缓冲或无缓冲操作的 ostream 类最适合于连续的文本模式输出。 基类 ios 的所有功能包含在 ostream 中。 如果构造 ostream 类的对象，则必须指定构造函数的 streambuf 对象。

ofstream 类支持磁盘文件输出。 如果需要一个只输出的磁盘，请构造 ofstream 类的对象。 在构造 ofstream 对象时或在调用该对象的 open 成员函数时，可以指定 ofstream 对象是否接受二进制或文本模式数据。 许多格式设置选项和成员函数适用于 ofstream 对象，包括基类 ios 和 ostream 的所有功能。

如果在构造函数中指定文件名，则在构造对象时，将自动打开该文件。 另外，可以在调用默认构造函数后使用 open 成员函数。

类似于运行时函数 sprintf_s，ostringstream 类支持向内存中的字符串进行输出。 若要通过使用 I/O 流格式设置在内存中创建一个字符串，请构造 ostringstream 类的对象。

Player和Enemy都是从Entity类派生出来的，然而对于派生类与基类之间的强制类型转换有一定的规则。

  1. 向上转型(Upcasting)：即派生类转化为基类，这在C++中是安全的。例如，你创建了一个 Player对象，并将其赋给了一个 Entity类型的指针 actuallyPlayer。由于 Player 是 Entity 的派生类，它可能具有比 Entity更多的数据和方法，但其基本结构必定符合 Entity 类。所以这样的向上转型是允许且安全的。
  2. 向下转型(Downcasting)：即基类转化为派生类，在C++中这是不安全的。例如，你尝试将一个 Entity类型（实际上是 Enemy对象）的指针 actuallyEnemy 转换为 Player 类型的指针，这就可能会出现问题。因为 Enemy和 Player 类可能有完全不同的数据和方法，一个 Enemy 对象不能完全贴合 Player 类的结构，因此这样的类型转换会导致错误。

请注意，当一次赋值于union的其中一个成员后，其他成员变得不可访问。联合体的大小由其最大的成员决定。 然而，使用联合体需要额外谨慎，因为由于与结构体的内存模型不同，我们需要手动确认存储的数据类型及其正确性。如果不小心将一种数据类型存入联合体，然后将其取出时当作另一种数据类型使用，那么就可能遇到问题，因为内部的字节表示根据解释的数据类型可能有所不同。如果数据类型不匹配，那么解释的数据可能就是无意义的。

例如，当使用 std::sort(vec.begin(), vec.end(), std::greater<int>()) 对 {3, 2, 5, 4, 1} 进行排序时，排序算法会在需要比较两个元素时调用 std::greater<int>() 的 operator()。这个函数对象会比较两个元素的大小，如果第一个元素大于第二个元素，则返回 true，否则返回 false。这样就实现了按照降序排序的功能。

除了greater类似的还有

  1. std::less: 它是默认的比较器，用于升序排序。与 std::greater 相反，它会比较两个元素，如果第一个元素小于第二个元素，则返回 true。
  2. std::greater_equal: 它用于降序排序，类似于 std::greater，但当两个元素相等时也会返回 true。
  3. std::less_equal: 用于升序排序，类似于 std::less，但当两个元素相等时也会返回 true。
  4. std::equal_to: 它用于检查两个元素是否相等。
  5. std::not_equal_to: 用于检查两个元素是否不相等。

  • 还可以接受lambda表达式，例如

    1. if (a == 2) return false;：当元素a等于2的时候，返回false。在sort函数中，当比较函数返回false，意味着第一个参数a应该排在第二个参数b的后面。所以所有等于2的元素都会被放在其他元素之后。
    2. if(b == 2) return true;：当元素b等于2的时候，返回true。即元素a应该排在元素b的前面。
    3. return a < b;：对于不等于2的元素，按照正常的从小到大的规则排序。

    因此，这个排序规则的主要功能是将所有的2都排列到最后，其他的元素则按照正常的从小到大的顺序排序。

    这样理解，a和b表示的是正在比较的两个数之间的位置关系，a在前b在后，所以默认升序排列也就是要求a<b因为 std::sort 的默认行为是实现升序排序，也就是由小到大排序。当我们提供一个自定义的比较函数（或者在案例中是一个Lambda表达式）来 std::sort 时，这个比较函数应当按照如下原则解释：
    如果 比较函数(a, b) 返回 true，那么元素 a 应当在元素 b 之前。换句话说，元素 a 应当被认为是 “小于” 元素 b。
    如果 比较函数(a, b) 返回 false，那么元素 a 不应当在元素 b 之前。也就是元素 b 在排序后应该位于元素 a 之前。
    所以在例子中，当 a 等于 2 时，比较函数返回 false，意味着元素 a（也就是 2） 不应该在 b 之前，因此 2 就被排到所有其他数的后面。同理，当 b 等于 2 时，比较函数返回 true，表示 a（不论其值）都应该在 b（也就是 2）前面，从而确保2总是排在后面。其余情况下，比较函数 a < b 则确保其他数按照正常的升序排列

效率

时间复杂度为O(NlogN)。在实际应用中，它是非常高效的。当然了，如果你要处理的数据非常大，比如上亿数量级，那可能就得考虑其他的排序策略，如归并排序，基数排序等。

.h&.hpp

在 C++ 中，.cpp 文件通常用作源代码文件，包含程序的实现部分，而 .h 或 .hpp 文件则作为头文件，包含类的声明和模板定义。

.hpp 文件是 .h 文件的一个变体，通常包含模板类或模板函数的定义。由于 C++ 模板需要在编译时完全可见（即声明和定义都需要在同一编译单元中），因此 .hpp 文件经常用于包含模板的实现，这样就可以在包含 .hpp 文件的地方直接使用模板。

这里是 .hpp 文件的一些特点：

  • Header Plus Plus：.hpp 是 Header Plus Plus 的缩写，表示它是一个 C++ 头文件 。
  • 模板实现：.hpp 文件通常包含模板类或函数的定义，这样模板的用户只需包含一个文件即可 。
  • 减少编译次数：由于模板的实现直接包含在 .hpp 文件中，它们会被编译到包含它们的 .cpp 文件中，从而减少了编译次数 。
  • 适合开源库：.hpp 文件非常适合用于开源库，因为它们简化了模板的使用和分发 。

总的来说，.hpp 文件是 C++ 中的一个约定，用于存放模板的声明和定义，以便于模板的分发和使用

模板编程

在 C++ 中，模板类的声明和实现需要放在同一个文件中的原因与模板的编译机制有关。模板是一种泛型编程的工具，它允许程序员编写代码模板，然后编译器会根据模板生成特定类型的代码。这个过程称为模板实例化。

当编译器遇到模板使用的地方，比如创建一个模板类的对象或调用一个模板函数，它需要访问模板的定义来生成代码。如果模板的声明和实现分别位于不同的文件中（通常是 .h 和 .cpp 文件），编译器在处理 .cpp 文件时无法预知模板参数的具体类型，因此无法生成相应的对象代码。这意味着链接器在最终链接程序时无法找到模板实例的定义，从而导致链接错误123。

为了确保模板的每个实例都能被正确地实例化，模板的声明和实现必须放在同一个文件中，通常是头文件（.h 或 .hpp）。这样，每当模板被使用时，编译器都能找到必要的信息来生成模板实例的代码

可能我们需要先复习一下程序的生成：

  1. 预处理阶段：预处理器处理 .cpp 文件中的预处理指令，如 #include，将头文件的内容插入到 .cpp 文件中。这一步不涉及编译，只是文本替换。
  2. 编译阶段：
    • 每个 .cpp 文件被编译成对象文件（.o 或 .obj）。此时，编译器只检查函数声明和其他符号的声明，而不需要定义。
    • 如果一个 .cpp 文件中的代码调用了另一个 .cpp 文件中的函数，编译器会在当前对象文件中生成一个“符号引用”。
  3. 链接阶段：
    • 链接器处理所有对象文件和库文件，解决符号引用，将它们合并成一个可执行文件。
    • 如果一个 .cpp 文件中的函数调用了另一个 .cpp 文件中的函数，链接器会在这一步找到被调用函数的定义，并“链接”它们。
  4. 模板处理：
    • 模板类或函数在编译阶段被处理。当编译器遇到模板实例化时，它会根据模板定义和实例化时提供的具体类型生成代码。
    • 模板的代码生成是按需进行的。如果多个 .cpp 文件实例化了相同的模板，则每个 .cpp 文件都会生成自己的实例化代码。
    • 在链接阶段，链接器会确保每个模板实例只保留一份，以避免重复。

总结来说，预处理器首先处理 #include 指令，编译器然后编译 .cpp 文件生成对象文件，链接器最后解决符号引用并生成可执行文件。模板类或函数的代码在编译阶段按需生成，并在链接阶段去除重复的实例化

模板的代码并不会直接被生成，而是在传入具体的类型或值之后，将模板类型与模板值全部替换为对应的版本，这样就生成了一个普通的类或函数

std::thread

std::thread 是 C++11 引入的标准库类，用于创建和管理线程。线程允许多个函数并发执行，从而提高程序的性能和响应能力。以下是对 std::thread 的详细讲解：

  • 线程：一个线程代表一个独立的执行序列，可以与其他线程并发执行。
  • 主线程：程序启动时的初始线程。
  • 子线程：由主线程或其他线程创建的线程。

线程可以帮助我们实现代码的并行执行，例如有std::cin.get()方法，会阻塞当前进程来等待输入空格，如果我们想在阻塞的时候做些别的事，应该怎么做呢，当前进程已经被阻塞，所以就考虑多线程来帮助我们完成另一件事.

在main中通过thread创建一个线程，指定其执行体为DoWork函数体，并立即开始了这个线程。从创建了新的线程开始，当前的程序就是多线程的，就有了主线程和子线程的区分，main函数的流程也就是原本的运行流程就是主线程，正在执行DoWork的线程就是子线程

由于各线程之间的运行速度有很大的随机性，所以为了：

  • 确保线程完成执行

    ​ 当你创建一个子线程时，它会在后台运行。如果主线程在子线程完成之前结束，程序将终止，子线程可能没有机会完成其任务。

  • 资源管理

    ​ 每个线程都占用一定的系统资源（如内存、句柄等）。

  • 数据一致性

    ​ 在多线程程序中，子线程可能会修改共享数据。如果主线程在子线程完成之前访问这些数据，可能会导致数据不一致或未定义行为。使用 join 可以确保主线程在子线程完成后再访问共享数据

  • 避免未定义行为

    ​ 如果主线程在子线程完成之前结束，程序可能会出现未定义行为。

  • 线程同步

    ​ 在某些情况下，你可能需要确保多个线程按特定顺序执行。

需要线程管理 join()来阻塞主线程来等待对应线程的结束，类似与多进程中的wait，用来阻塞进程等待子进程的结束，避免出现僵尸进程。

  • 使用 join 可以确保主线程等待子线程完成执行。可以确保这些资源在线程完成后被正确释放。如果不调用 join 或 detach，程序可能会导致资源泄漏。确保主线程在子线程完成后再访问共享数据。确保主线程在子线程完成后再继续执行，从而避免未定义的行为。确保一个线程在另一个线程完成后再开始执行。

join 和 detach

  • join：等待线程完成执行。
  • detach：将线程与当前线程分离，允许线程在后台运行。

​ 虽然同一个进程的多个线程共享进程的栈空间，但是，每个子线程在这个栈中拥有自己私有的栈空间。所以，线程结束时需要回收资源。

线程安全

线程安全（Thread Safety）是指在多线程环境中，代码或对象能够被多个线程同时访问而不会导致任何预期之外的行为或竞态条件。换句话说，线程安全的代码可以被多个线程同时使用，而不会出现数据不一致或程序崩溃等问题。

线程安全的核心概念

  1. 原子性：一个操作或一系列操作要么全部执行，要么全部不执行，不会出现中间状态。例如，银行转账操作必须是原子的，要么转账成功，要么转账失败，不会出现部分成功的情况。
  2. 可见性：一个线程对共享变量的修改，其他线程能够立即看到。由于现代处理器和编译器的优化，线程对变量的修改可能不会立即对其他线程可见。使用适当的同步机制可以确保可见性。
  3. 有序性：程序执行的顺序按照代码的先后顺序执行。编译器和处理器可能会对指令进行重排序，以提高性能，但这种重排序不会影响单线程程序的正确性。在多线程环境下，需要确保指令的执行顺序对所有线程一致。

如何保证线程安全

1. 互斥同步

使用互斥锁（如 std::mutex）来保护共享数据，确保同一时刻只有一个线程可以访问共享数据。

2. 原子操作

使用 std::atomic 提供的原子操作，可以避免使用锁，从而提高性能。

3. 条件变量

使用条件变量（std::condition_variable）来实现线程间的等待和通知机制。

线程安全的常见问题

  1. 数据竞争：多个线程同时访问和修改共享数据，导致数据不一致。
  2. 死锁：多个线程相互等待对方释放资源，导致程序无法继续执行。
  3. 活锁：线程不断地改变状态，但无法继续执行。
  4. 饥饿：某些线程长时间无法获得所需的资源，导致无法执行。

<chrono>库是一个非常有用的工具。<chrono>库提供了一组强大的时间处理工具，允许你以类型安全和高精度的方式执行时间操作。

Date and time utilities - cppreference.com

这个库引入了三个主要的时间点类型：

  1. std::chrono::system_clock：代表系统时钟，通常是系统范围内的实时时钟。
  2. std::chrono::steady_clock：代表单调递增的时钟，不受系统时钟的影响，适用于计时器和性能测量。
  3. std::chrono::high_resolution_clock：可能是system_clock或steady_clock，具体取决于实现，提供高分辨率的时间测量。

high_resolution_clock 在不同标准库实现之间实现并不一致，尽量不要使用。通常它只是 std::chrono::steady_clock 或 std::chrono::system_clock 的别名——它是 system_clock 时不是单调的（即时间能后退），但具体实现取决于库或配置。例如：对于 gcc 的 libstdc++ 它是 system_clock ，对于 MSVC 它是 steady_clock ，而对于 clang 的 libc++ 它取决于配置。

通常用户应该直接使用 std::chrono::steady_clock 或 std::chrono::system_clock 代替 std::chrono::high_resolution_clock ：对时长度量使用 steady_clock ，对壁钟时间使用 system_clock 。

<chrono>库还提供了不同的时间持续类型，用于表示时间段的各种单位，例如秒、毫秒、微秒和纳秒。这些类型包括：

  • std::chrono::duration：表示时间段的基本类型。
  • std::chrono::seconds：表示秒。
  • std::chrono::milliseconds：表示毫秒。
  • std::chrono::high_resolution_clock::time_point：这是高分辨率时钟的时间点类型。
  • **std::chrono::duration**：这是一个浮点型的时间段类型，用于存储开始和结束之间的时间差。

计时器Timer类

  • m_StartTimepoint 是一个 std::chrono::high_resolution_clock 时间点，使用 high_resolution_clock 确保我们有最高的可能时间分辨率。
  • Timer() 构造函数在 Timer 对象创建时被调用，它将 m_StartTimepoint 设置为当前的时间点。
  • ~Timer() 析构函数在 Timer 对象生命周期结束时被调用，它调用 Stop() 函数来计算时间差并进行输出。
  • Stop() 函数首先获取当前的时间点 endTimepoint；然后转换初始时间点和结束时间点为自 epoch 开始的微秒数（即1970年1月1日00:00:00至今的微秒数）；接着，计算这两个时间点的时间差 duration；这个时间差用微秒表示，转换为毫秒存入 ms；最后，将 duration（微秒）和 ms（毫秒）输出。
  • 注意到 Stop() 函数并没有更改 m_StartTimepoint 的值，也就是说，如果在同一个 Timer 对象上多次调用 Stop() 函数，那么每次调用都会输出从对象创建开始至当前调用的时间。

这种使用方式称为“资源获取即初始化”（RAII）模式。

__debugbreak()是一个特殊的 Microsoft Visual C++ intrinsic 函数，用于激活调试器的断点。当代码执行到 __debugbreak()时，如果有调试器附加到进程，debugger 将暂停执行，否则，Windows 将显示一个错误对话框，询问用户是否要附加调试器。 你可以用 __debugbreak()来在你的代码中插入一个断点，这可以在调试复杂问题或调试不易复现的 bug 时非常有用。

在 Release 模式的代码中通常不会使用 __debugbreak()，因为它会通过异常方式中断程序的正常执行。 需要注意的是，__debugbreak()是 Microsoft Visual C++特有的，不是标准 C++。
